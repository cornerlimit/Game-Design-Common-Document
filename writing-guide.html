📝 策划功能模块文档撰写指南

本指南说明如何撰写一份标准的 Gameplay 功能模块策划文档。
遵循统一的文档结构，确保信息完整、便于评审和开发落地。

← 返回模块总览

一、文档整体结构

每份功能模块策划文档由 一个 Excel/在线表格文件 组成，包含 两个 Sheet 页：

Sheet 1
Request
（需求总览）
→
Sheet 2
Design
（详细设计）
核心原则：Request 页让所有人快速了解"这个模块是什么、谁负责、分几块、什么优先级"；Design 页让开发和QA了解"具体怎么做、规则是什么、边界在哪里"。

2.1 各区块详细说明

① 模块标题（第1行）

格式：[项目代号]_[模块名].[子系统名]

示例：FSO_Kili.EvolutionSystem、ProjectX_Combat.MeleeSystem

命名规则统一使用英文，用点号 . 分隔层级，便于后续在 TAPD/Jira 中建立对应的 Feature。

② 概览 Overall（第2-6行）

用 3-5 条 bullet point 概述模块的核心目标和能力范围。

  • 每条以 - 开头
  • 描述的是"做什么"而非"怎么做"
  • 读者看完后应该能一句话复述这个模块的用途
避免在概览中写实现细节——那是 Design 页的内容。概览应聚焦于"用户/玩家能感知到什么"。

③ 责任人表（第9-15行）

明确每个岗位的负责人。包含以下角色行：

角色说明是否必填
类别填写本文档对应的模块名称（如"战斗系统"、"AI系统"）必填
PM项目经理/制作人选填
开发主程序负责人必填
策划模块策划负责人必填
美术美术资产负责人（原画/模型/场景等）按需填
动画动画资产负责人按需填
音频音频资产负责人按需填

④ TAPD Feature Link & Feature Priority（第16-17行）

TAPD Feature Link：关联的项目管理平台链接，便于跳转查看开发进度。

Feature Priority：模块整体优先级（P0-P4），决定排期优先度。

⑤ 需求细分表（第19行起）

将模块拆解为可独立开发的 子功能/子任务，每行一个。

列名说明示例
TASK COMPONENT子任务所属组件/团队（如GPP=Gameplay Programming）GPP
开发需求子功能的简要描述，编号与 Design 页的章节号对应1 AIGC对话功能
Priority子功能优先级（P0最高-P4最低）P3
TAPD对应的 TAPD 任务链接链接
更新时间最近一次更新日期2026-04-09
预估时间预估开发工时/工期5d
关键规则：需求细分表中的编号（1, 2, 3...）必须与 Design 页的一级章节号一一对应。这是两个 Sheet 之间的"桥梁"。

3.1 各区块详细说明

① 文档标题（第1行）

格式：[模块名].[子系统名].[功能名]，与 Request 页标题保持对应关系。

② 版本历史表

记录文档的每次变更，便于追溯。

列名说明
版本号从1开始递增
内容本次变更的简要描述（如"添加文档"、"增加XX功能"、"修改XX规则"）
作者/日期变更人/变更日期
备注补充说明（如"评审后修改"）
每次修改文档内容后，必须在版本历史中新增一行记录。禁止直接修改而不记录。

③ 1 Overview（概述章节）

与 Request 页的概览内容保持一致，但可以稍微展开。使用 1.1, 1.2, 1.3... 的编号。

④ 2+ 功能章节（核心内容）

从第 2 章开始，逐个展开 Request 页中列出的子功能。编号与 Request 页的需求编号一一对应。

四、多级编号规则（重要）

Design 页使用 多级数字编号，最多5级，用缩进体现层级关系：

2 剧情生成 ← 一级标题（对应Request中的子功能编号）
2.1 获得脚本 ← 二级：功能拆分
2.1.1 本地上传 ← 三级：具体方式
2.1.1.1 字数限制... ← 四级：规则/参数
2.1.1.1.1 可在settings里配置... ← 五级：补充说明
*2.1.1.2 敏感字词检测 ← 带 * 号表示待确认/TODO项
2.2 脚本生成对话 ← 二级：另一个功能分支
编号规则总结：
· 一级 (X)：模块的核心功能块，对应 Request 页的行
· 二级 (X.X)：功能块的拆分子项
· 三级 (X.X.X)：具体实现方式或流程步骤
· 四级 (X.X.X.X)：规则、参数、配置说明
· 五级 (X.X.X.X.X)：补充细节、边界情况
· * 前缀：表示待确认/未定/需讨论的条目

七、写作规范与常见问题

✅ 推荐做法

  • 每个规则/参数都要有 明确的数值或范围（如"冷却时间3秒"，而非"冷却时间适中"）
  • 使用 条件-结果 句式描述逻辑（如"当玩家HP≤30%时，触发低血量警告"）
  • 复杂逻辑用 表格/流程图/状态图 辅助说明
  • 可变的配置参数用 project settings可配置 标注
  • 不确定的内容用 * 前缀 标记为TODO
  • 设计决策要写明 理由（如"选择方案A而非B，因为..."）

❌ 常见问题

  • 模糊描述："伤害适当降低" → 应改为 "伤害降低30%"
  • 漏写边界：未说明空值/极端情况的处理方式
  • 只写正常流程：缺少异常流程/失败处理/中断恢复的说明
  • Request和Design不对应：需求细分表的编号和Design章节号对不上
  • 不更新版本历史：修改了内容但没有在版本历史中记录
  • 把实现方案写成需求："用协程实现延迟" → 应改为 "技能有2秒释放延迟"

📋 自查清单（提交前检查）

  • ☐ Request 页标题、概览、责任人是否完整？
  • ☐ 需求细分表的编号是否和 Design 页对应？
  • ☐ 每个子功能是否都有 Priority？
  • ☐ Design 页版本历史是否已更新？
  • ☐ 所有数值参数是否给了具体值或配置说明？
  • ☐ 边界情况和异常流程是否覆盖？
  • ☐ TODO/待确认项是否都用 * 标注？
